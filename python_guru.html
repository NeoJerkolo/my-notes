List — під капотом

Динамічний масив (C array). В пам'яті — суцільний блок покажчиків на PyObject. При перевищенні capacity Python алоцює новий блок із надлишком (~12.5%). Саме тому append() — амортизована O(1).

import sys
lst = []
for i in range(10):
    lst.append(i)
    print(sys.getsizeof(lst))
# Побачиш стрибки: 56→88→88→88→88→120→120→...
# growth: 0→4→8→16→25→35→46→58→72→88 slots
append() є амортизованою O(1). Для справжньої O(1) на обох кінцях → collections.deque.

lst.reverse()           # O(n) — перевертає in-place

⚠️ sort() vs sorted(): sort() мутує і повертає None. sorted() повертає новий список. Класична помилка: lst = lst.sort() → lst стане None!

Timsort — як він працює

Гібрид mergesort + insertion sort. Шукає вже відсортовані "runs" (мін. 32-64 елементи), потім merge. Стабільний. O(n) на вже відсортованих даних, O(n log n) у гіршому.

# key= функція викликається O(n) разів, не O(n log n)
# → використовуй operator замість lambda — швидше
from operator import itemgetter, attrgetter
lst.sort(key=itemgetter(1))          # по другому елементу
lst.sort(key=attrgetter('name'))     # по атрибуту
lst.sort(key=itemgetter(1, 0))       # мультисортування

# Schwartzian transform (DSU) — обчисли ключ один раз
decorated = [(key(x), x) for x in lst]
decorated.sort()
result = [x for _, x in decorated]

# Slice assignment — мутує БЕЗ нового об'єкта!

Dict — під капотом (Compact Hash Table)

CPython 3.6+ використовує compact dict: окремий масив індексів + щільний масив (key, hash, value). До 3.6 порядок не зберігався. Collision resolution — open addressing з pseudo-random probing.

# Як відбувається lookup d[k]:
# 1. hash(k) → хеш значення
# 2. hash % capacity → початковий слот
# 3. Якщо slot зайнятий іншим ключем → probing
# 4. load factor > 2/3 → resize (подвоєння capacity)

# Цікавий факт: рівні хеші = проблема
hash(1) == hash(1.0) == hash(True)  # True!
d = {1: 'a', 1.0: 'b'}
# → {1: 'b'} — тільки 1 ключ! 1.0 перезаписав
# Бо: 1 == 1.0 AND hash(1) == hash(1.0)
Для ключа dict необхідно: (1) реалізувати __hash__, (2) реалізувати __eq__, (3) якщо a == b то hash(a) == hash(b). Hashable: str, int, float, tuple*, frozenset. НЕ hashable: list, dict, set.

Set — під капотом

По суті dict без values. Hash table з open addressing. Елементи мають бути hashable. Немає дублікатів, немає гарантованого порядку (хоча CPython 3.7+ зберігає порядок вставки для dict, але НЕ для set).

# Literals vs constructor — важлива відмінність!
s = {1, 2, 3}          # set literal
s = set()              # ПОРОЖНІЙ set ← правильно
s = {}                  # ПОРОЖНІЙ dict! ← пастка!
s = frozenset([1,2])  # незмінний, hashable

# set з дублікатами — автоматично видаляє
s = {1, 1, 2, 2, 3}   # → {1, 2, 3}

# set як "seen" для дедуплікації з порядком
seen = set()
unique = [x for x in lst if x not in seen and not seen.add(x)]
set значно швидший за list для перевірки in: O(1) vs O(n). Якщо робиш багато перевірок — конвертуй список у set першим ділом.

s1 - s2  == s1.difference(s2)
s1 ^ s2  == s1.symmetric_difference(s2)
# ^ = елементи що є РІВНО в одному з двох множин

# Порівняння

s1 < s2   # строге підмножина: s1 ⊂ s2
s1 >= s2  == s1.issuperset(s2)   # s1 ⊇ s2
s1.isdisjoint(s2)               # True якщо s1 ∩ s2 = ∅

Tuple — під капотом

Незмінний масив покажчиків. Менший за list (немає overhead на resize). Python кешує tuple до 20 елементів — повторне створення може не алоцювати нову пам'ять!

import sys
sys.getsizeof((1,2,3))  # ~72 bytes
sys.getsizeof([1,2,3])  # ~88 bytes

# Tuple з 1 елементом — КОМА обов'язкова!
t = (1,)    # tuple ✓
t = (1)     # int! ← пастка
t = 1,      # tuple теж! (без дужок)

# Кешування малих tuple
a = (1, 2)
b = (1, 2)
a is b  # True! (CPython оптимізація)
Unpacking — всі форми
# Базовий unpacking
a, b, c = 1, 2, 3
a, b = b, a          # swap без temp — атомарно!

# Extended unpacking (3.0+)
first, *rest = [1, 2, 3, 4]   # first=1, rest=[2,3,4]
*init, last = [1, 2, 3, 4]   # init=[1,2,3], last=4
a, *_, z = [1,2,3,4,5]      # a=1, z=5, _=[2,3,4]
a, (b, c) = 1, (2, 3)       # вкладений unpacking

# В for loops
for a, *b in [(1,2,3), (4,5,6)]:
    print(a, b)  # 1 [2,3], 4 [5,6]
Чому tuple hashable, а list — ні?

Hash об'єкта має бути незмінним протягом його lifetime. List мутабельний → hash міг би змінитись → порушилась би інваріантність dict/set. Tuple незмінний → hash стабільний. Але є нюанс:

# Tuple з mutable елементом — НЕ hashable
t = ([1, 2], 3)
hash(t)  # TypeError: unhashable type: 'list'

# Лексикографічне порівняння tuple
(1, 2) < (1, 3)    # True — перша різниця
(1,) < (1, 2)      # True — коротший менший
(1, 2) == (1, 2)  # True

tupleгетерогенні дані (координата, запис БД, кортеж повернення), константи, ключі dict, return multiple values
listгомогенна колекція елементів що змінюється, черга, стек

String — під капотом (PEP 393)

c is d  # False (CPython-specific, не гарантовано)

            for i in range(times):
                try: return func(*a, **kw)
                except exceptions as e:
                    if i == times-1: raise
        return wrapper
    return decorator
